## Source Framework Overview

A 4-prompt framework for designing a personalized second brain system. The framework enforces a strict sequence: purpose discovery, stack selection, build guide, stress test. The core principle is that architecture is universal but configuration must be personal.

**Key quote:** "The architecture is the same for everyone -- capture, classify, file, confirm, digest, fix. But the stack that actually ships depends on where you already live, what's falling through the cracks, and how much you're willing to maintain."

---

## 1. Universal Architecture Pattern

The source defines a 6-step closed-loop architecture that applies to all second brain implementations:

```
capture --> classify --> file --> confirm --> digest --> fix
   |                                                    |
   +<--------------------------------------------------+
                    (closed loop)
```

### Step Definitions

| Step | Purpose | Observable Output |
|------|---------|-------------------|
| **Capture** | Raw input enters the system | Record created in inbox |
| **Classify** | AI determines type, destination, metadata | Classification result with confidence score |
| **File** | Item stored in correct destination bucket | Record in one of 4 buckets (people/projects/ideas/admin) |
| **Confirm** | User receives acknowledgment of capture | Confirmation message in capture channel |
| **Digest** | Periodic synthesis of accumulated captures | Weekly/scheduled summary with themes and open loops |
| **Fix** | User corrects misclassification | Record moved, inbox log updated |

### ALMA Mapping of the 6 Steps

| Step | Current ALMA Component | Status | Gap |
|------|----------------------|--------|-----|
| Capture | `alma.extraction.AutoLearner` (partial) | Exists but agent-scoped, not user-scoped | Needs MCP `alma_capture_thought` tool |
| Classify | `alma.extraction.LLMFactExtractor` | Extracts facts but not destination-bucket classification | Needs classification pipeline with bucket routing |
| File | `alma.storage` (7 backends) | Storage exists, no automatic routing | Needs destination bucket abstraction layer |
| Confirm | `alma.events.emitter` + `alma.events.webhook` | Pub/sub exists, webhook delivery exists | Needs user-facing confirmation channel |
| Digest | Nothing | Not implemented | Needs weekly review synthesis engine |
| Fix | Nothing | Not implemented | Needs reclassification MCP tool + inbox log update |

## 2. Prompt-to-ALMA Component Mapping

### 2.1 Prompt 1: Purpose Discovery --> `alma init --open-brain`

Prompt 1 forces users to answer 7 questions before any tools are selected. This maps directly to ALMA's quickstart onboarding flow.

**Extracted Decision Framework:**

| Input | What It Determines | ALMA Config Output |
|-------|-------------------|-------------------|
| What's falling through the cracks | Primary use case | `purpose` field in config |
| What I've tried before | Anti-patterns to avoid | `excluded_categories` in DomainSchema |
| Where friction kills me | Pipeline focus (capture/retrieval/follow-through) | Feature flag priorities |
| What I stop doing every week | Success metric | `success_criteria` in config metadata |
| What "working" looks like | Observable outputs (3 max) | Digest/delivery configuration |
| Information types (2-3) | Entity types to enable | `entity_types` in Personal Brain schema |
| Cost of being wrong | Reliability tier | `reliability_tier` in config |

**Key Design Principle:** "If any answer is blank or vague, ask up to 6 clarifying questions (all at once) and STOP. Do not proceed until inputs are complete."

**ALMA Implementation Implication:** The `alma init --open-brain` CLI must be an interactive questionnaire, not a silent config generator. It should refuse to generate config until all required inputs are collected.

**Outputs Generated by Prompt 1:**

1. Primary use case (one sentence)
2. 3 observable outputs that make it worth building
3. 2 things the system should NOT try to do (scope control)
4. Operating mode: `always-on` | `scheduled` | `session-based`
5. Reliability tier: `good-enough` | `needs-receipts` | `full-audit-trail`
6. The one weekly task it replaces

### 2.2 Prompt 2: Stack Selection --> ALMA Backend Selection + Config Generation

Prompt 2 maps user reality to tool choices. For ALMA, this means selecting the right storage backend, embedder, and delivery mechanism.

**Extracted Reality Constraints:**

| Constraint | ALMA Equivalent | Config Key |
|-----------|----------------|------------|
| Where I already live (communication) | Source tools (Slack, Discord, etc.) | `capture.source` |
| Where I already live (notes/docs) | Not applicable (ALMA is the storage) | N/A |
| Automation I already use | Integration layer | `integrations` |
| AI access | Embedder + LLM for extraction | `embedder`, `extraction.llm` |
| Technical comfort | Config complexity level | `setup_mode: simple|standard|advanced` |
| Time for setup | MVP vs full build | `build_tier: mvp|standard|full` |
| Budget | Backend selection (free vs paid) | `storage.backend` selection |
| Devices | Capture interface selection | `capture.channels` |
| Privacy/compliance | Storage location, encryption | `storage.encryption`, `storage.local_only` |
| Captures per day | Performance tier | `storage.performance_tier` |
| Latency tolerance | Sync vs async processing | `capture.mode: sync|async|batch` |
| Where capture happens most | Client priority | `capture.primary_client` |

**Key Design Principle:** "Only recommend tools from MY allowed list. Do not suggest tools I've refused." ALMA should surface only compatible backends based on user constraints.

### 2.3 Prompt 3: Build Guide --> Capture Pipeline + Metadata Extraction + MCP Tools

Prompt 3 generates a runnable build guide with copy-paste assets. This is the densest section for ALMA implementation.

**Extracted Build Architecture:**

#### Destination Buckets (max 4 + inbox log)

| Bucket | Description | ALMA Entity Type Mapping |
|--------|-------------|--------------------------|
| **People** | Relationship notes, contact context | `person` entity (new) |
| **Projects** | Active work items, deadlines | `project` entity (new) |
| **Ideas** | Creative ideas, insights, research | `idea` entity (new) |
| **Admin** | Errands, logistics, administrative tasks | `task` entity (from general schema) |
| **Inbox Log** | All captures with pointers to destinations | `capture_log` (new, acts as audit trail) |

#### Max 5 Required Fields Per Bucket

This constraint maps directly to ALMA's `EntityType.attributes` pattern. The source framework limits required fields to 5 per bucket to reduce friction:

#### Classification Prompt Pattern

The build guide requires a "full classification prompt in the exact schema your storage layer needs." This is the core of the metadata extraction pipeline.

**Required prompt output schema:**

```json
{
  "destination": "people | projects | ideas | admin",
  "confidence": 0.0-1.0,
  "title": "short title",
  "extracted_fields": {
    "field1": "value1",
    "field2": "value2"
  },
  "people_mentioned": ["Name1", "Name2"],
  "action_items": ["action1"],
  "deadline": "ISO date or null"
}
```

#### Confidence Threshold + "Needs Review" Bucket

**Critical pattern:** When classification confidence is below a threshold, items go to a "Needs Review" bucket instead of being filed. The user then manually classifies them.

```
if confidence >= threshold:
    file_to_bucket(destination)
    send_confirmation()
else:
    file_to_needs_review()
    send_review_request()
```

**ALMA mapping:** The `alma.confidence` module already has `ConfidenceSignal` with scores. The gap is that this confidence system is currently backward-looking (strategy assessment), not forward-looking (classification confidence for new captures).

#### Fix Flow Pattern

The "fix flow" enables users to correct misclassifications:

```
User sends: "fix: people"
System:
  1. Moves last capture from current bucket to "people" bucket
  2. Updates inbox log entry with correction
  3. Sends confirmation: "Moved to People. Updated log."
```

**ALMA mapping:** This requires:
- A new MCP tool: `alma_reclassify_thought`
- An inbox log data structure (audit trail)
- Event emission on reclassification (`MEMORY_RECLASSIFIED` event type)

#### Definition of Done (Closed-Loop Test)

The source defines 6 checkboxes that must ALL pass before the system is considered working:

1. A capture creates a record in the correct destination bucket
2. An inbox log entry is created with a pointer back to the original capture
3. A confirmation message is sent to the capture channel
4. A low-confidence path exists (needs review bucket or flag)
5. A fix flow updates the log and refiles to the correct destination
6. A digest runs on schedule OR a session checklist exists

**ALMA implementation:** This becomes an integration test suite for the Open Brain feature.

### 2.4 Prompt 4: Stress Test --> Quality Gates and Test Phase

Prompt 4 identifies failure modes before building. This maps to ALMA's testing strategy.

**5 Most Likely Failure Modes:**

1. Malformed AI response (LLM returns invalid JSON)
2. Missing permission or disconnected integration
3. Low-confidence classification (ambiguous input)
4. Simple capture that works perfectly (happy path)
5. Fix correction that refiles and updates log

**ALMA Testing Mapping:**

| Test Type | Source Description | ALMA Test |
|-----------|-------------------|-----------|
| Failure injection: Malformed AI | LLM returns broken JSON | `test_extraction_handles_malformed_llm_response` |
| Failure injection: Missing permission | Integration disconnected | `test_capture_handles_storage_unavailable` |
| Failure injection: Low confidence | Ambiguous classification | `test_low_confidence_routes_to_review` |
| Normal: Simple capture | Happy path end-to-end | `test_capture_classify_file_confirm_e2e` |
| Normal: Fix correction | Reclassification | `test_fix_flow_refiles_and_updates_log` |

**5 Test Message Patterns:**

```python
TEST_MESSAGES = [
    # 1. Simple capture (clear classification)
    "Met with Sarah today. She's interested in the API project.",

    # 2. Ambiguous item (tests confidence threshold)
    "Need to think more about the pricing thing.",

    # 3. Person mention (tests people extraction)
    "Marcus prefers async communication and hates morning meetings.",

    # 4. Project with deadline (tests deadline extraction)
    "Website redesign due by March 15. Waiting on design assets from the team.",

    # 5. Fix correction (tests reclassification)
    "fix: projects",
]
```

**Key Stress Test Question:** "Which single component is the highest-risk dependency?" For ALMA Open Brain, this is the LLM extraction pipeline -- if the LLM is unavailable or returns garbage, nothing classifies.

**The One Habit:** "The one habit that will make or break this system" -- This maps to capture consistency. The framework acknowledges that no system survives without the user actually capturing. ALMA should optimize for minimal capture friction.

## 3. Design Patterns for ALMA Open Brain

### 3.1 Operating Modes

| Mode | Description | ALMA Implementation |
|------|-------------|-------------------|
| **Always-on** | Captures process immediately, digests on schedule | Real-time event processing via `EventEmitter` + cron digest |
| **Scheduled** | Captures batch-process at intervals | Batch processing in `capture.mode: batch` config |
| **Session-based** | User runs capture/review manually | CLI commands or MCP tool invocation |

**Recommendation:** ALMA should default to `always-on` for MCP-connected clients (captures via tool calls are instant) and `session-based` for CLI users.

### 3.2 Reliability Tiers

| Tier | Description | ALMA Implementation |
|------|-------------|-------------------|
| **Good enough** | Items may misfile occasionally, user fixes manually | Default confidence threshold (0.6), no audit trail |
| **Needs receipts** | All captures logged, corrections tracked | Inbox log enabled, `MemoryEventType.CREATED` events persisted |
| **Full audit trail** | Every operation logged, immutable history | Full event sourcing via `alma.events`, webhook to external system |

**ALMA mapping to existing infrastructure:**
- `good-enough`: Use `alma.storage` directly, no event persistence
- `needs-receipts`: Enable `alma.events.emitter` with storage mixin
- `full-audit-trail`: Enable `alma.events.webhook` + `alma.observability` (OpenTelemetry)

### 3.3 MVP-First Approach

The source framework mandates a 30-minute MVP on day 1:

```
MVP (day 1):  capture --> classify --> file --> confirm --> inbox log
Full (day 2+): + fix flow + digests + extensions
```

**ALMA quickstart mapping:**

```bash
# Day 1: MVP
alma init --open-brain
# Generates: config, schema, basic MCP tools
# User tests: 1 capture, verify it classifies and stores

# Day 2: Add fix flow + digest
alma open-brain enable fix-flow
alma open-brain enable weekly-digest
```

### 3.4 The 3 Operating Rules

The framework prescribes exactly 3 enforceable rules (not advice):

1. **One capture rule:** "Capture the thought within 60 seconds or let it go"
2. **One fix rule:** "Fix misfiles within 24 hours or accept the classification"
3. **One maintenance rule:** "Review digest weekly or the system decays"

**ALMA enforcement mechanism:**
- Capture rule: Not enforceable by software (user behavior)
- Fix rule: ALMA could auto-promote "Needs Review" items after 24 hours (configurable)
- Maintenance rule: ALMA could send reminders via webhook if digest is unread for 7+ days

### 3.5 Scope Control Pattern

Prompt 1 explicitly asks for "2 things this system should NOT try to do." This maps to ALMA's `DomainSchema.excluded_categories`:

```python

        "capture_patterns",
        "relationship_management",
        "project_tracking",
        "idea_development",
        "administrative_tasks",
    ],
    excluded_categories=[
        # User-defined during alma init --open-brain
```

## 4. Gap Analysis: Framework vs Current ALMA

### Critical Gaps (Must Address)

| Gap | Framework Has | ALMA Status | Priority |
|-----|--------------|-------------|----------|
| **Capture tool** | Natural language capture to structured record | `AutoLearner` is agent-scoped, not user-scoped | P0 |
| **Classification pipeline** | LLM classifies into buckets with confidence | `LLMFactExtractor` extracts facts but does not route to buckets | P0 |
| **Inbox log** | Append-only record of all captures with destination pointers | Nothing -- events exist but no queryable log | P0 |
| **Needs Review bucket** | Low-confidence items held for manual classification | `ConfidenceSignal` exists but no routing based on score | P1 |
| **Fix flow** | User command to reclassify + update log | Nothing | P1 |
| **Digest / weekly review** | Scheduled synthesis of accumulated captures | Nothing | P1 |
| **Personal Brain schema** | 4 buckets with max 5 fields each | No personal brain schema (6 schemas exist, none personal) | P0 |
| **Onboarding questionnaire** | 7-question interactive setup | No `alma init` CLI at all | P1 |

### Secondary Gaps (Important for Adoption)

| Gap | Framework Has | ALMA Status | Priority |
|-----|--------------|-------------|----------|
| **Confirmation delivery** | Message sent back to capture channel | `EventEmitter` + `WebhookDelivery` exist but need personal brain integration | P2 |
| **Operating mode config** | always-on / scheduled / session-based | No concept of operating modes in config | P2 |
| **Reliability tier config** | good-enough / needs-receipts / full-audit-trail | Observability exists but not tiered | P2 |
| **Test message suite** | 5 standard test messages for validation | No built-in self-test for Open Brain | P2 |
| **Parsing fallback plan** | Explicit handling for malformed LLM responses | `LLMFactExtractor` has basic error handling but no "Needs Review" fallback | P2 |

### Things ALMA Already Has That the Framework Lacks

|----------------|---------------|

| Multi-agent memory sharing | Single-user only, no agent collaboration |

| Trust scoring | No trust layer for retrieved memories |

## 5. ALMA Integration Recommendations

### For the Capture Pipeline Squad

**Action Items:**

1. **Create `alma_capture_thought` MCP tool** that accepts natural language input and returns a structured capture record with classification, confidence score, and destination bucket.

2. **Implement classification pipeline** as a new module `alma.capture.classifier`:
   ```python
   class ThoughtClassifier:
       def classify(self, content: str) -> ClassificationResult:
           """Classify raw text into destination bucket with confidence."""
           # Returns: destination, confidence, extracted_fields, people, action_items
   ```

3. **Add inbox log data structure** -- either as a new memory type in `alma.types` or as a separate table/collection. Must support:
   - Append-only entries
   - Pointer to destination record
   - Correction history (for fix flow)
   - Timestamp and source tool

4. **Implement "Needs Review" routing** -- when classification confidence is below threshold (configurable, default 0.6), route to a review queue instead of a destination bucket.

### For the Metadata Extraction Squad

**Action Items:**

1. **Extend `LLMFactExtractor`** to output the classification schema required by the capture pipeline:
   ```json
   {
     "destination": "people | projects | ideas | admin",
     "confidence": 0.85,
     "title": "Meeting with Sarah about API project",
     "people_mentioned": ["Sarah"],
     "action_items": ["Follow up on API project"],
     "deadline": null,
     "topics": ["api", "project"]
   }
   ```

2. **Add person extraction** -- the current `ExtractedFact` has `condition`, `strategy`, `category`, and `domain` fields but no `people` field. Person extraction is critical for the "People" bucket.

3. **Add deadline extraction** -- parse dates from natural language ("due by March 15", "next Friday", "end of Q2").

4. **Add parsing fallback plan** -- when LLM returns invalid JSON:
   - Attempt JSON repair (strip markdown fences, fix trailing commas)
   - If repair fails, route to "Needs Review" with raw text preserved
   - Log the failure for diagnostics

### For the Weekly Review Squad

**Action Items:**

   - Gap analysis

2. **Create `alma_weekly_review` MCP tool** that triggers synthesis on demand or via schedule.

3. **Define digest delivery** -- the framework supports multiple delivery modes:
