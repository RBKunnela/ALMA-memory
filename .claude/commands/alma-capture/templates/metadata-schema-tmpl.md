# Metadata Schema Template

**Purpose:** Define the structured metadata schema for captured thoughts. This schema is used by the metadata extraction pipeline and all downstream tools.

---

## Schema

```json
{
  "type": "decision | person_note | insight | meeting | task | idea | reference | lesson",
  "topics": ["topic1", "topic2"],
  "people": ["Person Name"],
  "action_items": ["action 1"],
  "importance": 1,
  "source_tool": "claude | chatgpt | cursor | slack | manual",
  "captured_at": "2026-03-03T10:30:00Z",
  "context": "optional context about when/where this was captured"
}
```

---

## Field Descriptions

### type (required)

The classification of the thought. Exactly one of the following:

| Type | Description | Example Trigger |
|------|-------------|-----------------|
| `decision` | A decision that was made, with reasoning | "We decided to...", "After weighing options..." |
| `person_note` | Information about a person -- preferences, style, context | "Sarah prefers...", "John mentioned he..." |
| `insight` | A realization, aha moment, or non-obvious connection | "I just realized...", "The key insight is..." |
| `meeting` | Notes from a meeting, call, or conversation | "Met with...", "In today's standup..." |
| `task` | Something that needs to be done | "Need to...", "TODO:", "Remember to..." |
| `idea` | A creative idea, proposal, or possibility to explore | "What if we...", "Idea:", "Could we..." |
| `reference` | Factual information to remember -- a link, fact, config value | "The API endpoint is...", "Password for..." |
| `lesson` | A lesson learned from experience -- what worked, what did not | "Lesson learned:", "Next time I should...", "Mistake:" |

### topics (required, list of strings)

Keywords or phrases describing what this thought is about. Guidelines:
- 1-5 topics per thought (aim for 2-3)
- Use lowercase, natural language phrases
- Prefer specific over generic ("react hooks" over "programming")
- Topics should help with future retrieval

### people (optional, list of strings)

Names of people mentioned in or relevant to the thought. Guidelines:
- Use the name as it appears in the text
- Include full names where available, first names otherwise
- Do not include generic references ("the team", "my manager") unless a specific person is identifiable
- Empty list if no specific people are mentioned

### action_items (optional, list of strings)

Concrete actions that need to be taken. Guidelines:
- Start each with a verb ("Update the plan", "Send email to...")
- Be specific enough to act on
- Only include explicit or strongly implied actions
- Empty list if the thought has no actionable items

### importance (required, integer 1-5)

How important this thought is for future reference:

| Score | Label | Description | Example |
|-------|-------|-------------|---------|
| 1 | Trivial | Passing thought, low future value | "The coffee here is good" |
| 2 | Context | Useful background, might be relevant later | "The office wifi password is..." |
| 3 | Notable | Worth remembering, good to have searchable | "Team velocity has been improving" |
| 4 | Important | Significant decision, insight, or relationship context | "We chose PostgreSQL over MongoDB because..." |
| 5 | Critical | Career-defining, high-stakes, or foundational | "The board approved the pivot to AI-first" |

### source_tool (optional, string)

Which tool or interface was used to capture this thought:
- `claude` -- captured from Claude / Claude Code
- `chatgpt` -- captured from ChatGPT
- `cursor` -- captured from Cursor IDE
- `slack` -- captured from Slack
- `manual` -- manually entered via CLI or direct API

Default: `"manual"` if not specified.

### captured_at (auto-generated, ISO-8601 string)

The timestamp when the thought was captured. Auto-generated by the capture pipeline. Always in UTC with timezone info.

Format: `"2026-03-03T10:30:00Z"`

### context (optional, string)

Free-form text providing additional context about when, where, or why this thought was captured. Examples:
- "During quarterly planning meeting"
- "While reviewing PR #142"
- "After reading the Stripe blog post on API design"

---

## Validation Rules

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| type | string | Yes | Must be one of 8 valid types |
| topics | list[str] | Yes | 1-5 items, non-empty strings |
| people | list[str] | No | 0-10 items, non-empty strings |
| action_items | list[str] | No | 0-10 items, non-empty strings |
| importance | int | Yes | 1-5 inclusive |
| source_tool | string | No | One of: claude, chatgpt, cursor, slack, manual |
| captured_at | string | Auto | ISO-8601 with timezone |
| context | string | No | Max 500 characters |

## Edge Cases

### Short input (< 20 words)

**Input:** "PostgreSQL connection string for prod: host=db.example.com port=5432"

Expected: type=reference, importance=2. Short inputs often lack signals for topics/people -- extract what you can, use `["database", "configuration"]` as topics.

### Multi-topic input

**Input:** "Great meeting with the design team about the new dashboard. Also discussed the Q2 hiring plan and budget allocation for cloud infrastructure."

Expected: type=meeting, topics should include all three themes. Importance depends on the significance of the decisions.

### No people mentioned

**Input:** "The codebase has too much coupling between the storage and retrieval layers. Need to introduce a proper abstraction boundary."

Expected: people=[], action_items=["introduce abstraction boundary between storage and retrieval"]. Do not hallucinate people.

### Ambiguous type

**Input:** "Talked to investors today. They want to see 10x growth in 6 months. This changes everything about our roadmap."

This could be meeting, insight, or decision. Prefer `meeting` when the input describes a conversation, even if it contains insights.

### Multiple people

**Input:** "Lunch with Sarah, Jake, and Maria. Sarah is leaving the company in March. Jake is taking over her responsibilities. Maria wants to move to the platform team."

Expected: people=["Sarah", "Jake", "Maria"]. All three should be captured.
